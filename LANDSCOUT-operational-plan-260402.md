## 1. DATA LAYER — From Fabricated to Real

Everything in LandScout flows from data. The entire app is currently powered by 75 seed records generated by `prisma/seed.ts`. Making it real means replacing that pipeline.

### 1.1 Database Migration: SQLite → PostgreSQL

| Item | Current | Needed |
|------|---------|--------|
| Database | SQLite file (local) | PostgreSQL on Neon (Vercel Marketplace) |
| Data at runtime | Static JSON files exported at build time | Live Prisma queries against Neon |
| Writes | Optimistic (fake responses, nothing persists) | Real writes to PostgreSQL |
| API routes | Read from `/data/*.json` files | Query Prisma directly |

**What this unlocks:** Every mutation in the app (deal stage changes, activity logging, new parcels from crawling) actually persists. Without this, nothing else is truly "real."

**Work required:**
1. Provision Neon PostgreSQL via Vercel Marketplace (5 min)
2. Update `prisma/schema.prisma` datasource from `sqlite` to `postgresql`
3. Revert all 6 API routes from file-based reads back to Prisma queries
4. Remove `scripts/export-data.ts` and `data/` directory
5. Run `prisma db push` against Neon
6. Run seed script against Neon for initial data
7. Test all pages against live DB

**Difficulty:** 2/5
**Dependencies:** None — this is the foundation everything else builds on.

### 1.3 Scanner Job System

The orchestrator that runs all adapters on a schedule.

| Item | Current | Needed |
|------|---------|--------|
| Scanner class | Not present in v2 (was in v1) | Port from v1 `lib/jobs/scanner.ts` |
| Scheduling | Vercel Cron configured in v1 | Add `vercel.json` cron or use Vercel Cron API |
| Scan tracking | ScanRun + ScanRunSource tables exist | Wire up to real scan executions |
| Error handling | Schema supports error logging | Log per-source errors, retry logic |
| Rate limiting | Config says 6 req/min per source | Implement delay between requests |

**Scanner pipeline per run:**
1. Create `ScanRun` record (status: running)
2. For each enabled source:
   a. Create `ScanRunSource` (status: running)
   b. Get adapter instance
   c. Build search params from AdminConfig (states, acreage range, etc.)
   d. Call `adapter.search(params)` with rate limiting
   e. For each result:
      - Deduplicate: find existing parcel by APN+County+State or lat/lon proximity
      - Create or update Parcel
      - Create or update Listing (link to parcel and source)
      - Calculate FitScore
   f. Update `ScanRunSource` with counts
3. Update `ScanRun` with totals

**Trigger options:**
- Weekly cron via Vercel: `POST /api/admin/scan` every Sunday 2:00 UTC
- On-demand via Admin UI button (not yet built)
- Per-source manual trigger

**Difficulty:** 3/5
**Dependencies:** 1.1 (PostgreSQL), 1.2 (at least 2 adapters working)

### 1.4 Deduplication Engine

When the same 500-acre parcel appears on LandWatch AND Lands of America AND Hall & Hall, we need to merge them into one Parcel with three Listings.

| Method | Priority | How |
|--------|----------|-----|
| APN match | Highest | Same APN + County + State = definite match |
| Geographic clustering | High | Lat/lon within 0.1 miles + acreage within ±10% = probable match |
| Title similarity | Medium | Fuzzy string matching on listing title + same county = possible match |

**What exists now:** Schema supports multiple listings per parcel. Dedup logic needs to be written.

**Difficulty:** 3/5
**Dependencies:** 1.2 (adapters producing data)
